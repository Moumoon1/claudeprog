# PRD Issue Rules

本文件用于细化 PRD / 原型审查时的问题判断标准。

## 1. 正式问题

正式问题指：基于当前 PRD / 原型内容，已经可以判断会影响逻辑闭环、页面设计、交互方案或后续落地的问题。

常见类型：

### 1.1 前后矛盾
- 同一个规则在不同段落描述不一致
- 同一个状态在不同页面表现不一致
- 文案描述和原型表现冲突
- 按钮名称与实际动作不一致

### 1.2 流程断层
- 用户完成上一步后，不知道下一步去哪里
- 页面之间缺少返回、取消、保存、完成等闭环
- 提交后结果页、反馈、状态变化不明确
- 流程只描述入口，没有描述出口

### 1.3 状态遗漏
只在影响交互闭环时指出，不机械要求所有状态。

重点关注：
- 空态
- 加载态
- 成功态
- 失败态
- 禁用态
- 权限不足
- 无结果
- 提交中
- 保存失败
- 删除确认

### 1.4 页面职责混乱
- 一个页面同时承担查看、编辑、配置、分析等过多任务
- 主任务被次要信息打断
- 页面入口与页面内容不匹配
- 列表、详情、编辑职责边界不清

### 1.5 操作预期不一致
- 用户以为是保存，实际是提交
- 用户以为是立即生效，实际需要审核
- 用户以为可撤销，PRD 未说明撤销逻辑
- 用户以为可多选，后续规则却像单选

### 1.6 原型不可落地
- 页面内容在真实数据下明显撑不开
- 表格字段过多但没有横向滚动或折叠策略
- 文案过长但没有截断规则
- 弹窗承载过复杂流程
- 移动端/窄屏下关键操作难以成立

### 1.7 设计系统冲击
- 引入新组件但未说明必要性
- 打破现有组件使用规则
- 需要新的复杂状态样式
- 现有组件无法承载该交互

---

## 2. 待确认问题

待确认问题指：当前信息不足，但这个缺口会影响设计判断或交互方案。

可以输出待确认问题的情况：

- 规则缺失会影响页面结构
- 状态缺失会影响交互闭环
- 数据来源不明会影响展示方式
- 权限规则不明会影响页面可见性
- 操作结果不明会影响反馈设计
- 原型表现和 PRD 描述不一致，需要确认以哪个为准

不建议输出待确认问题的情况：

- 只是背景信息没写
- 只是用户画像不够完整
- 只是业务目标没有展开
- 不影响当前页面设计判断
- 只是理论上可能存在的异常

---

## 3. 优化建议

优化建议指：不影响主流程成立，但可能提升体验、降低理解成本或减少设计复杂度。

常见类型：
- 文案更清晰
- 操作路径可缩短
- 信息层级可优化
- 模块可合并
- 次要操作可降噪
- 状态反馈可更明确

注意：
- 优化建议不要冒充正式问题
- 不要为了凑数量输出大量泛泛优化建议
- 优先输出和设计落地直接相关的建议

---

## 4. 优先级判断

### P0
必须在设计前确认或修复，否则设计方案无法成立。

例如：
- 主流程不闭环
- 页面职责冲突严重
- 关键状态未定义
- 前后规则明显矛盾
- 原型和 PRD 表达互相冲突

### P1
会影响设计方案或导致返工，需要尽快确认。

例如：
- 关键操作反馈不明确
- 重要分支状态缺失
- 页面结构可能需要调整
- 组件使用方式不清楚
- 用户路径存在绕路或断点

### P2
不影响主流程，但会影响体验或细节完善。

例如：
- 文案语义可优化
- 局部信息层级不够清晰
- 次要状态可补充
- 交互反馈可增强

---

## 5. 问题描述原则

问题要具体，不要空泛。

好的写法：
- “保存后没有说明是否返回列表页，导致设计无法确定成功反馈和页面去向。”
- “PRD 中写支持批量操作，但原型只提供单选入口，选择机制前后不一致。”
- “详情页同时承担查看和编辑任务，但没有区分只读态和编辑态，页面职责不清。”

不好的写法：
- “建议完善流程。”
- “建议补充异常情况。”
- “这里体验可以优化。”
- “需要补充背景。”
