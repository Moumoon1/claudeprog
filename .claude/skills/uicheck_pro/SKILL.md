# Goal

1. 找出可稳定确认的正式问题
2. 记录疑似问题
3. 文档输出为表格
4. 双图必须框同一问题对象
5. 图片必须先保存到 workspace

# Image Mapping Rules

- 用户指定 dev/design 时不得交换
- 未指定才可推断

# Issue Rules

差异分三类：

- 正式问题
- 疑似问题
- 不纳入问题（动态数据 / 滚动 / 状态栏 / 轻微渲染）

优先关注：
- 模块缺失 / 顺序错误
- 骨架 / 层级 / 对齐问题
- 关键组件 / 颜色 / 尺寸
- 信息架构错误
- 标签 / 图例 / 字段规则错误

---

## Design Completeness Rule（解决漏查）

必须以设计稿为基准逐模块检查：

- 设计稿中每个模块 / 元素，都必须在开发页确认是否存在
- 若存在于设计稿但开发页不存在 → 必须输出正式问题
- 不得因为未发现差异而跳过模块

---

## Data & Chart Exclusion Rule（合并版）

以下差异默认不作为问题：

- 金额、数量、百分比、转化率
- 日期、时间、用户名、头像、排名、状态
- 图表数据值、报表数值
- 数据驱动的图形变化（宽度 / 高度 / 面积 / 曲线）

对于图表 / 漏斗：

只检查：
- 结构
- 标签
- 单位
- 图例
- 样式 / 布局 / 层级

不检查：
- 数值一致性
- 图形形态（漏斗宽度、柱高等）

只有以下情况才算问题：

- 字段缺失
- 标签错误
- 单位错误
- 图例错误
- 结构 / 样式 / 层级错误
- 设计稿明确固定形态但未实现

# Screenshot Rules（重写核心）

## CropRegion vs Box（新架构）

坐标输出必须拆分为两组：

- **CropRegion**（裁图区域）：稍大于问题区域，保留上下文，让用户看到问题所在环境
  - 例：top=0.0, bottom=0.15, left=0.0, right=1.0（截取页面顶部 15%）
- **Box**（红框区域）：问题元素本身的精确位置，框尽量小而准
  - 例：top=0.02, bottom=0.10, left=0.1, right=0.5（框住某个标题/按钮）

规则：
- 元素级问题 → Box 尽量小，只框出问题的具体元素
- 模块级问题 → Box 框住整个出问题的模块
- 细节级问题 → Box 框住对应细节区域
- Box == CropRegion 表示整个裁图区域都是问题（大范围结构问题），此时画淡色框
- 不合格的 Box：框整张图（top=0 bottom=1 left=0 right=1）→ 必须缩小

## Element Matching Rule（解决框歪）

双图框选必须基于“元素匹配”，不得基于位置：

流程：

1. 先在设计稿识别元素（按钮 / 卡片 / 图表等）
2. 在开发页按语义查找对应元素
3. 找到后再框选

---

## No Position Projection Rule

禁止：
- 把设计稿框直接映射到开发图同一位置

原因：
- 尺寸不同
- 布局偏移
- 滚动差异

---

## Matching Failure Handling（解决空框）

若开发页找不到对应元素：

- 不得框空白
- 不得猜位置
- 必须标记为“组件缺失”

截图方式：

- 设计稿：精确框元素
- 开发页：框模块区域或标注“未实现”

---

## 基础规则（保留）

- 两边必须框同一对象
- 框选必须与问题描述一致
- 一条问题必须有双图
- 不得用模糊区域代替

---

# Problem Description Rule

只允许：

- 问题

不允许：

- 标题
- 位置
- 长分析

---

# Document Output Rules

结构固定：

1. 标题
2. 摘要
3. 页面总览
4. 问题表格

表头：

- 序号 / 问题描述 / 状态 / 截图 / 日期 / 优先级 / 走查人 / 跟进人

---

# Folder Mapping Rule

- 按文件名匹配 dev / design
- 不匹配的必须提示
- 每一对独立走查

---

# Restrictions（精简版）

- 不得交换 dev/design
- 不得把动态数据当问题
- 不得把数据驱动图形当问题
- 不得混淆正式 / 疑似
- 不得框错对象
- 不得框空白
- 不得用位置映射代替元素匹配
- 不得把“数据差异”脑补为“规则问题”
